# Story 1.6: Performance Testing & Overhead Validation

## Story Statement

**As a** quality engineer,
**I want** to benchmark agent execution time with and without instrumentation enabled,
**so that** I can confirm overhead is <5% and identify any bottlenecks.

---

## Acceptance Criteria

1. **Baseline Measurement**
   - Run @dev, @qa, @architect agents 10 times each WITHOUT instrumentation
   - Record execution duration, memory usage, CPU time for each run
   - Calculate average, median, p95, p99 latencies
   - Establish baseline metrics for comparison

2. **Instrumented Measurement**
   - Run @dev, @qa, @architect agents 10 times each WITH instrumentation enabled
   - Record same metrics (duration, memory, CPU)
   - Calculate statistics (average, median, p95, p99)

3. **Overhead Calculation**
   - Calculate overhead percentage: `(avg_instrumented - avg_baseline) / avg_baseline * 100`
   - Assert overhead <5% for each agent
   - Assert with 95% confidence (statistical significance)
   - Report: `@dev: 1.2% overhead`, `@qa: 0.8% overhead`, `@architect: 1.5% overhead`

4. **Component Breakdown**
   - Measure overhead contributed by each component:
     - Agent event hooks (~0.5%)
     - Skill hooks (~0.3%)
     - Event serialization (~0.2%)
     - Async send/retry (~0.2%)
     - Sanitization (~0.3%)
   - Identify which components contribute most overhead
   - Document any surprising results

5. **Test Harness**
   - Create standalone test harness (not dependent on Dashboard or Auditor)
   - Run agents on standard task/scenario (same task for baseline and instrumented)
   - Ensure no external I/O affects measurements (mock Collector if needed)
   - Allow repeated runs for statistical confidence

6. **Report Generation**
   - Generate comprehensive performance report with:
     - Baseline statistics (avg, median, p95, p99)
     - Instrumented statistics
     - Overhead percentages per agent
     - Component contribution breakdown
     - Recommendations for optimization (if overhead high)
   - Save report as CSV and JSON for analysis

## Dev Notes

### Architecture Context
- **Source:** docs/architecture.md#Non-Functional Requirements, #Performance Optimization
- **NFR1:** Instrumentation overhead <5%
- **Test Approach:** Benchmark baseline vs instrumented runs
- **Statistical Confidence:** 95% confidence (small sample size, use t-test if needed)

### Measurement Strategy
- **Baseline:** Agents run without instrumentation (environment variable INSTRUMENTATION_ENABLED=false)
- **Instrumented:** Agents run with instrumentation (INSTRUMENTATION_ENABLED=true)
- **Same Task:** Both baseline and instrumented use identical input/task
- **Isolation:** Mock Collector endpoint to avoid network variability
- **Warm-up:** 1-2 warmup runs before measurement (to stabilize memory allocation)
- **Sample Size:** 10 runs per configuration (enough for statistical analysis)

### Metrics to Capture
- **Execution Duration:** High-resolution timer (process.hrtime.bigint() or console.time)
- **Memory Usage:** process.memoryUsage() at start and end (heap used, external, rss)
- **CPU Time:** user time, system time (may not be available on all platforms)
- **Event Count:** Track how many events emitted (for cost analysis)

### File Locations
- **Test Harness:** `packages/aiox-instrumentation/tests/performance.test.ts`
- **Sample Task:** Fixed task for @dev, @qa, @architect (or use existing test tasks)
- **Report Template:** `packages/aiox-instrumentation/tests/performance-report-template.md`
- **Results:** `packages/aiox-instrumentation/tests/results/` (CSV, JSON files)

### Testing Requirements
- Integration test: Baseline measurement (@dev, @qa, @architect)
- Integration test: Instrumented measurement
- Analysis: Statistical comparison (average overhead)
- Report generation: Save results in multiple formats

## Notes

- This story is critical gate; overhead >5% blocks progress to Dashboard/Auditor
- If overhead unexpectedly high, investigate:
  - Synchronous I/O in event path (shouldn't be)
  - Regex overhead in sanitization (need optimization)
  - Database client initialization (should be pooled)
  - Large event payloads (consider sampling or truncation)
- Performance metrics establish baseline; future stories can compare against this
- Sample size (10 runs) may be small for statistical significance; increase if needed
- Warmup runs help stabilize memory allocation; ensure they're discarded before analysis
