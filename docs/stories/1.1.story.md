# Story 1.1: Implement Instrumentation Hooks in @dev Agent

## Story Statement

**As a** framework maintainer,
**I want** the @dev agent to emit lifecycle events (started, finished) to a collector,
**so that** I can observe when @dev begins and completes tasks and measure execution duration.

---

## Acceptance Criteria

1. **Event Emission on Startup**
   - When @dev is activated, an event `{event_type: "agent.started", agent_id: "dev", timestamp, input, execution_id}` is emitted to collector endpoint `POST /events`
   - Execution ID is a unique UUID generated per @dev activation
   - Input contains the task description passed to @dev
   - Emission happens before agent processing begins

2. **Event Emission on Completion**
   - When @dev completes or errors, an event `{event_type: "agent.finished", agent_id: "dev", timestamp, output, duration, status}` is emitted
   - Duration is calculated in milliseconds between start and finish
   - Status is one of: "success", "error", "partial"
   - Output contains the final result or error message

3. **Reliability & Fallback**
   - If collector is unavailable, events are queued locally in-memory with exponential backoff retry (max 3 attempts, 100ms → 200ms → 400ms)
   - Failure to send events does NOT interrupt @dev execution (fully asynchronous)
   - Events are not lost during normal operation

4. **Performance Constraint**
   - Measured overhead from instrumentation is **<1%** of baseline @dev execution time
   - Instrumentation hooks must be non-blocking (async)
   - No synchronous I/O in critical path

5. **Transparency**
   - No agent functionality is altered; instrumentation is transparent to end user
   - User sees no instrumentation messages in normal output
   - @dev behavior is identical with and without instrumentation

## Dev Notes

### Architecture Context
- **Source:** docs/architecture.md#CLI Instrumentation, #High Level Architecture
- **Event Schema:** JSON with required fields: event_type, agent_id, execution_id, timestamp
- **Transport:** HTTP POST to Collector at endpoint `/events` (Bearer token auth)
- **Non-blocking Requirement:** Use Node.js EventEmitter or async queue pattern
- **Retry Strategy:** Exponential backoff (100ms, 200ms, 400ms) with max 3 attempts

### Technical Constraints
- **Performance:** <1% overhead (baseline must be established in Story 1.6)
- **Language:** JavaScript/TypeScript (AIOX core language)
- **Integration Point:** @dev agent activation hook (before execution)
- **Dependencies:** None beyond Node.js built-ins (EventEmitter, setTimeout)
- **Testing:** Must be measurable via Story 1.6 performance benchmarking

### Implementation Approach
1. Create `packages/aiox-instrumentation/src/event-emitter.js` - Core event emission engine
2. Hook into @dev activation (insert registration call)
3. Emit `agent.started` event on activation
4. Hook into @dev completion (wrap result handling)
5. Emit `agent.finished` event on completion
6. Implement local queue + retry logic for collector unavailability
7. Verify no overhead via micro-benchmarks

### File Locations
- **Instrumentation Package:** `packages/aiox-instrumentation/`
- **Event Emitter:** `packages/aiox-instrumentation/src/event-emitter.js`
- **Hook Registration:** `.aiox-core/agents/dev.md` (add initialization call)
- **Tests:** `packages/aiox-instrumentation/tests/event-emitter.test.js`

### Testing Requirements
