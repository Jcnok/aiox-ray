# Story 1.3: Implement Skill Execution Event Capture

## Story Statement

**As a** framework maintainer,
**I want** to capture skill execution events (skill.executed) including skill name, parameters, duration, and result,
**so that** I can identify which skills consume time and which fail frequently.

---

## Acceptance Criteria

1. **Skill Event Emission**
   - When any skill (CLI built-in or custom) is invoked, a `skill.executed` event is emitted
   - Event payload contains: `{event_type: "skill.executed", agent_id, skill_name, parameters (sanitized), duration_ms, status, result_summary}`
   - Skill name uniquely identifies the skill being executed

2. **Event Linking**
   - Skill events are linked to parent agent execution via `execution_id` (same as agent.finished event)
   - Collector can correlate all skills from a single agent execution

3. **Parameter Sanitization**
   - Skill parameters are sanitized (secrets removed) before emission (Story 1.5 handles full sanitization)
   - Common sensitive fields filtered: API keys, tokens, passwords
   - Sanitization adds minimal overhead

4. **Non-blocking Execution**
   - Skill execution is NEVER blocked waiting for event emission
   - Event emitted asynchronously after skill completes
   - Skill error does not prevent event emission

5. **Event Delivery**
   - Event ingestion latency is <100ms in 95th percentile
   - If Collector unavailable, event is queued locally (consistent with agent events)

6. **Status Tracking**
   - Status field is one of: "success", "error", "timeout", "skipped"
   - result_summary contains brief output or error message (sanitized, not full stack trace)

## Dev Notes

### Architecture Context
- **Source:** docs/architecture.md#Events, #Data Models
- **Event Type:** skill.executed (added to enum in event schema)
- **Linking:** Via execution_id from parent agent.started/finished
- **Payload:** skill_name (string), parameters (JSONB, sanitized), duration_ms (integer), status (enum), result_summary (string)
- **Transport:** Same HTTP POST to Collector as agent events

### Technical Constraints
- **Sanitization Scope:** Story 1.3 does basic filtering; comprehensive sanitization in Story 1.5
- **Performance:** Event emission must not add measurable overhead to skill execution
- **Integration Point:** Hook into AIOX skill execution pipeline (likely in CLI core)
- **Result Summary:** Max 500 chars to avoid storing large outputs

### Implementation Approach
1. Create middleware/hook for skill execution (wrap AIOX skill dispatcher)
2. On skill start: Record timestamp and skill_name
3. On skill end: Calculate duration, capture status, sanitize parameters and result
4. Emit skill.executed event to EventEmitter (from Story 1.1)
5. Ensure skill execution continues regardless of event emission status

### File Locations
- **Skill Hook:** `packages/aiox-instrumentation/src/skill-hook.js` (new)
- **Hook Registration:** AIOX CLI skill dispatcher (location TBD)
- **Tests:** `packages/aiox-instrumentation/tests/skill-hook.test.js`
- **Integration:** Collector will persist skill.executed events (Story 1.4)

### Testing Requirements
- Unit test: skill.executed event structure validation
- Unit test: Duration calculation accuracy
- Unit test: Parameter sanitization (basic filtering)
- Unit test: Skill error captured without propagation
- Integration test: Skill events linked to parent agent execution
- Performance test: Event emission doesn't block skill execution
