# Story 1.4: Build Collector Service & PostgreSQL Storage

## Story Statement

**As a** backend engineer,
**I want** a Node.js/Express collector service that receives events via POST /events, validates them, and persists to PostgreSQL,
**so that** events are stored durably and queryable.

---

## Acceptance Criteria

1. **HTTP Endpoint: POST /events**
   - Collector listens on configurable port (default 3001)
   - Accepts JSON event payload with Content-Type: application/json
   - Returns HTTP 201 with event_id on success
   - Validates event schema before persistence (required fields: event_type, agent_id, timestamp, execution_id)
   - Returns HTTP 400 with error details on validation failure
   - Rejects payloads >10KB (prevents abuse)

2. **Event Validation**
   - Validate required fields present and correct types
   - Validate event_type is in enum: agent.started, agent.finished, error.occurred, recovery.attempt, skill.executed, chain_of_thought
   - Validate agent_id is one of: dev, qa, architect, orchestrator
   - Validate timestamp is ISO 8601 format
   - Validate execution_id is UUID format
   - Invalid events logged and rejected; valid events persisted

3. **Database Persistence**
   - Events inserted into PostgreSQL `events` table
   - Schema matches architecture spec: event_id (UUID), event_type (string), agent_id (string), execution_id (UUID), timestamp (timestamp), duration_ms (integer), payload (JSONB), version (string)
   - Indices created on (agent_id, timestamp DESC), (execution_id), (event_type), (timestamp DESC) for query performance
   - Auto-generated event_id (UUIDs)

4. **Query API: GET /events**
   - Supports filters: ?agent_id=dev&start_time=2026-03-13T00:00Z&end_time=2026-03-13T23:59Z&event_type=agent.finished
   - Returns paginated results: ?limit=100&offset=0
   - Default limit: 100, max: 1000
   - Response: `{events: [...], total: N, limit: 100, offset: 0}`
   - HTTP 200 with events; HTTP 400 if invalid filters

5. **Authentication**
   - All endpoints require Bearer token in Authorization header
   - Token validated against environment variable COLLECTOR_TOKEN
   - Returns HTTP 401 if missing; HTTP 403 if invalid
   - Token validation has minimal overhead

6. **Performance**
   - Can handle 100+ events/second without queueing or loss under normal conditions
   - Query response time <100ms for typical filters
   - Database connection pooling configured (min 5, max 20 connections)

7. **Error Handling**
   - Invalid events logged with request context but don't crash server
   - Database errors logged; requests return HTTP 500 with generic error message (no DB details leaked)
   - Server gracefully handles database unavailability

8. **Health Check**
   - GET /health endpoint returns HTTP 200 with `{status: "ok"}` when operational
   - Used by monitoring/deployment systems

## Dev Notes

### Architecture Context
- **Source:** docs/architecture.md#Collector Service, #Backend Architecture, #Database Schema
- **Framework:** Express.js 4.18+ with TypeScript
- **Database:** PostgreSQL 14+ with JSONB support
- **Stack:** Node.js 18+, pg (node-postgres client), Pino (structured logging)
- **Port:** Configurable via environment variable PORT (default 3001)

### Database Setup
- **Schema DDL:** See docs/architecture.md#Database Schema for complete table definitions
- **Tables:** events (main), agents (metadata), executions (aggregated), cot_segments (CoT data)
- **Partitioning:** Events table partitioned by timestamp (30-day retention)
- **Indices:** (agent_id, timestamp DESC), (execution_id), (event_type), (timestamp DESC)
- **Cleanup:** Cron job to delete events older than 30 days

### Configuration
- **PORT:** Environment variable, default 3001
- **DATABASE_URL:** PostgreSQL connection string (e.g., postgresql://user:pass@localhost:5432/aiox_ray)
- **COLLECTOR_TOKEN:** Bearer token for authentication
- **LOG_LEVEL:** debug | info | warn | error (default: info)
- **NODE_ENV:** development | production

### File Locations
- **Collector App:** `squads/aiox-ray/collector/src/index.ts` (Express app entry)
- **Routes:** `squads/aiox-ray/collector/src/routes/events.ts` (POST /events, GET /events)
- **Controllers:** `squads/aiox-ray/collector/src/controllers/eventsController.ts`
- **Services:** `squads/aiox-ray/collector/src/services/eventService.ts` (validation, persistence)
- **Database:** `squads/aiox-ray/collector/src/db/client.ts` (connection pool), `queries.ts` (SQL builders)
- **Middleware:** `squads/aiox-ray/collector/src/middleware/auth.ts`, `errorHandler.ts`, `requestLogger.ts`
- **Tests:** `squads/aiox-ray/collector/tests/integration/events-api.test.ts`

### Testing Strategy
- Unit tests: Event validation, error handling
- Integration tests: HTTP endpoints with real/mock database
- Test database: Use docker-compose with test PostgreSQL instance
