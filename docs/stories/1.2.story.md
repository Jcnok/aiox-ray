# Story 1.2: Extend Instrumentation to @qa and @architect Agents

## Story Statement

**As a** framework maintainer,
**I want** @qa and @architect agents to emit the same lifecycle events as @dev,
**so that** I have complete visibility into all core agent executions.

---

## Acceptance Criteria

1. **@qa Agent Events**
   - @qa emits `agent.started` event when activated (same schema as @dev)
   - @qa emits `agent.finished` event when completed (same schema as @dev)
   - Events are sent to same Collector endpoint with same authentication

2. **@architect Agent Events**
   - @architect emits `agent.started` event when activated
   - @architect emits `agent.finished` event when completed
   - Events follow identical schema and delivery mechanism

3. **Code Reuse**
   - No code duplication; both agents reuse EventEmitter from Story 1.1
   - Hook registration pattern is identical across all three agents
   - Single source of truth for event schema and transport logic

4. **Execution ID Traceability**
   - Each agent activation generates unique execution_id (UUID)
   - execution_id links related events from same agent run
   - Collector can correlate events by execution_id

5. **No Agent Functionality Altered**
   - @qa behavior unchanged; instrumentation is transparent
   - @architect behavior unchanged; instrumentation is transparent
   - Both agents produce identical output with/without instrumentation

6. **Measured Performance**
   - Instrumentation adds <1% overhead to each agent (same as @dev)
   - No blocking on event emission
